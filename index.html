The Sandbox style Metaverse - 2D world map

This is a more recent feature I worked on. The implementation was smooth, fast and fun. I took an existing world map asset from the Unity Asset Store, moulded it to fit design specifications and incorporated it into our project. Modifications include: profiling and fixing performance and memory leaks, grid and map style changes, region detection, generating grid data JSON files and fetching grid cell data with GraphQL. Like 'The Sandbox', the 'Lands' (cells on the grid) can be purchased as real estate. Our game is heavily blockchain and Play to Earn integrated with 'hero' and 'pet' NFTs and the ability to convert premium currency into our token and withdraw. We use the Polygon network, an Ethereum layer 2 solution.

'AddComponent' tracking and editor script

The performance of 'AddComponent' is quite bad, so we were looking for a way to avoid instances of this. I tracked wherever this happened at runtime, storing relevant hierarchy information in JSON files. I created an editor script to run through our prefabs and add the tracked components upfront. Developing a local tool and getting the feedback of it adding value is a great feeling. Developing a public app isn't as immediately rewarding but having something that can provide value to others or make money is different reward entirely.

Leaderboards

Leaderboards is one of the more popular features in our game. We have 70 leaderboards so scalability and reusability was a top concern. One pattern we created as a team was the use of a 'config' data struct which can be modified dynamically with extension methods and which is used for passing around instead having functions accepting long lists of parameters. Abstraction, scrolls with reusable elements, pagination, request and response, all very usual for me. If I were to refactor anything, I would try to merge separate leaderboard lookups into one location so the process of adding new leaderboard was more streamlined. I didn't make the backend for these leaderboards, but I've debugged and fixed a few bugs on the server-side code. We are running a Unity C# server, using an Amazon DynamoDB, and interacting with the client with Entitas Entities. We moved away from this server architecture for newer features. Now we make isolated services written in Go, using PostgreSQL, and interacting with the client with GraphQL. This architecture allows a service go down without taking everything else with it.

Weekly Riddle

I've done a few client-side GraphQL implementations and this is one of them. Using GraphQL includes an inconvenient code generation stage but then allows you to interact with data classes as you normally would in C#. I'm very comfortable with the Unity UI system and have done a lot myself in that regard. Those rewards at the bottom are tracked on the client and server using Entities from the Entity Component System, Entitas. The client can make a request to have an action performed on the server (e.g. claiming rewards). You declare an action handler, validating that the state is appropriate for the modification (the rewards aren't already claimed) and then modify the state (Components on Entities in the Context). The client will be rolled back to the server state it becomes 'out-of-sync'.
